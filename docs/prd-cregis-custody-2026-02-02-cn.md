# 产品需求文档 (PRD) - Cregis Custody

**作者:** Daniel  
**日期:** 2026-02-02

## 执行摘要
为受监管机构提供银行级加密货币托管服务，实现安全的资产管理、治理和合规性，并与企业财务系统无缝集成。作为银行和机构向其客户提供金融服务的基础设施。

## 成功标准
- 机构客户入驻数量
- 托管资产规模 (AUC)
- 合规审计通过率
- 客户留存率和满意度
- 企业客户入驻时间

## 用户旅程 (User Journeys)
### 4.1 企业资金管理 (原型: Everypay 平台)
**主要角色:** 企业财务主管
**接口:** Everypay 平台 (通过 Cregis 插件/API 集成)

**场景:** 一家跨国贸易公司使用 Everypay 管理跨境加密货币结算。
1.  **入驻与授权:** 财务主管登录 Everypay，启动 "连接托管"，并通过 **Cregis 财库单元创建授权页面** 授权 Everypay 代表其管理财库单元。
2.  **应收账款创建:** 财务主管在 Everypay 中为合同 #CT-2026-001 (`biz_id`) 创建一个 "应收账款"，标签为 `Procurement` (采购，`biz_type`)。Cregis 生成一个唯一的、一次性的 `RECEIVABLE` (应收) 地址。
3.  **归集与资金池:** 客户存入资金。Cregis 检测到存款，标记为 `CT-2026-001`，并自动将其从 `RECEIVABLE` 归集 (Sweep) 到 `PRIMARY` (主) 钱包。Everypay 收到 Webhook 并更新合同状态。
4.  **受策略控制的支付:** 财务主管发起供应商付款。Everypay 调用 Cregis API **创建一个特定的支付任务**。**交易详情与签名页面** (Cregis 提供的插件) 嵌入在 Everypay 中弹出，显示任务详情。财务主管 (和/或通过 **Cregis Task APIs** 获取待处理任务的其他授权客户) 必须使用插件对该任务进行密码学 **签名**，使其生效并在链上网络发布。
5.  **高级操作 (约束):** 复杂的资金操作 (锁定、释放、手动归集) 对 Everypay 隐藏。这些专业操作由 **机构管理员** 通过 **Cregis 托管客户门户** (见旅程 4.4) 独家管理。

### 4.2 加密支付处理 (原型: GlobalMall 演示)
**主要角色:** 电商商户
**接口:** GlobalMall (通过 Cregis API 集成)

**场景:** 一家在线电子产品零售商在结账时接受加密货币。
1.  **下单:** 购物者结账订单 #GM-8821 (`Order Id`)。GlobalMall 调用 Cregis 创建支付订单。
2.  **入金流程:** Cregis 分配一个 `PAYIN` (入金) 地址。购物者付款。Cregis 检测到资金，将其移动到 `PAYIN (Received)`，并触发 Webhook。GlobalMall 确认订单。
3.  **风险隔离:** 由于金额较大，风险引擎标记该交易。资金从 `PAYIN` 移动到 `QUARANTINE` (隔离区)。
4.  **释放与结算:** 合规官通过 **Cregis 托管客户门户** 审核并释放资金。Cregis 执行 `Release` (释放) -> `PRIMARY`。GlobalMall 收到结算通知。

### 4.3 个人托管 (原型: FinSecure 银行演示)
**主要角色:** 银行运营系统 / 终端用户
**接口:** FinSecure 移动应用 (后端与 Cregis 集成)

**场景:** 一家零售银行向其客户提供加密货币钱包。
1.  **用户激活:** 银行客户 Alice 在 FinSecure 应用中激活她的加密钱包。FinSecure 调用 Cregis 为用户 #FS-Alice-001 (`user_id`) 配置一个 `DEPOSIT_IN` (充值) 地址。
2.  **充值:** Alice 发送 BTC。Cregis 检测到充值进入 `DEPOSIT_IN`，自动归集 (`Pool`) 到 `PRIMARY`，并通知 FinSecure。FinSecure 更新 Alice 在应用中的余额。
3.  **提现:** Alice 请求提现。FinSecure 验证她的 2FA，然后调用 Cregis `Withdraw` API。
4.  **执行:** Cregis 分配资金 `PRIMARY` -> `WITHDRAW_OUT` (提现出口) -> 外部地址。

### 4.4 机构管理 (接口: Cregis 托管客户门户)
**主要角色:** 合规官 / 管理员
**接口:** Cregis 原生 Web 门户

**场景:** 高级治理和专业资金操作。
1.  **规则配置:** 管理员登录配置 "大额转账" 规则 (超过 $100k 需要 3/5 审批)。
2.  **签署:** 管理员使用 **财库单元规则配置签名页面** 对新策略进行密码学签名并激活。
3.  **专业资金操作:** 管理员/合规官登录执行敏感资金变动:
    *   **Release (释放):** 风险验证后批准从 `QUARANTINE` 释放资金。
    *   **Lock (锁定):** 手动冻结 `QUARANTINE` 中的资产以进行调查。
    *   **Pull (提取):** 执行直接的 `PRIMARY` -> 外部冷存储提现。
    *   **Allocate (分配):** 内部钱包之间的资金手动分配。
4.  **审计:** 管理员导出 "月度合规报告" 以验证所有操作。

### 4.5 系统运维 (接口: Cregis 管理控制台)
**主要角色:** 系统操作员
**接口:** Cregis 内部控制台

**场景:** 监控平台健康状况和完整性。
1.  **对账警报:** 系统标记某个 `PAYIN` 钱包的 Cregis 账本与区块链之间存在不匹配。
2.  **调查:** 操作员查看 "不同步" 交易列表，检查 `biz_id` 标签，并启动 **会计恢复** 功能。
3.  **解决:** 系统重新同步状态，确保持久账本与链上实况一致。

## 功能需求

### 通用
- 策略引擎: 每种交易类型可配置的审批链
- 审计追踪: 不可篡改、可导出的证据，具有保留规则
- 角色与权限: 最小权限原则，职责分离
- 事件机制: 用于状态变更的 Webhook 和内部事件
- 对账: 托管账本 ↔ 外部账本/系统
- 实时余额 API: 获取特定财库单元下所有账户 (PRIMARY, PAYIN 等) 的当前余额，支持按资产类型过滤

### 个人托管 (Individual Custody)
*   **账户结构:**
    *   **PRIMARY:** 主要资产持有钱包。
    *   **DEPOSIT_IN:** 指定的入金存款地址/钱包。
    *   **WITHDRAW_OUT:** 指定的提现处理钱包。
    *   **QUARANTINE:** 风险控制和审计检查的隔离区。
*   **操作与资金流:**
    *   **Deposit (充值):** 外部 → DEPOSIT_IN
    *   **Pool (归集):** DEPOSIT_IN → PRIMARY (自动归集)
    *   **Lock (锁定):** DEPOSIT_IN → QUARANTINE
    *   **Release (释放):** QUARANTINE → 外部 或 QUARANTINE → PRIMARY
    *   **Allocate (分配):** PRIMARY → WITHDRAW_OUT
    *   **Withdraw (提现):** WITHDRAW_OUT → 外部
    *   **Pull (提取):** PRIMARY → 外部 (直接提现)
*   **接口语义:**
        *   **创建账户:** 生成充值地址。
        *   **通知:** 充值通知。
        *   **请求:** 提现请求。
        *   **活动列表:** 获取带有 `user_id` 和 `biz_type` 标签的交易历史 (充值/提现)，用于第三方对账。
        *   **标签:** `user_id`, `biz_type`
    *   **策略定义:**
            *   **支持资产:** 定义该财库单元实例授权的具体加密货币 (如 BTC, ETH, USDT-ERC20)。
            *   **资金出金管理策略 (Fund Outbound Mgmt Policy):**
                *   **范围:** 控制 `Withdraw` 操作 (`WITHDRAW_OUT` → 外部)。
                *   **规则:**
                    *   **白名单:** 可选定义允许的外部目标地址。
                    *   **审批人:** 支持最多 4 层审批组。在每一层，每种支持的加密货币都与特定的交易金额限额挂钩。
            *   **资金池管理策略 (Fund Pool Mgmt Policy):**
                *   **范围:** 控制 `Pool`, `Allocate`, 和 `Pull` 操作 (主账户)。
                *   **规则:**
                    *   **流向强制:** 强制内部流向 (如 `DEPOSIT_IN` → `PRIMARY`)。
                    *   **白名单:** 可选的 `Pull` 操作外部目标地址。
                    *   **审批人:** 单一审批组。
            *   **资金风控管理策略 (Fund Risk Mgmt Policy):**
                *   **范围:** 控制 `Lock` 和 `Release` 操作 (隔离账户)。
                *   **规则:**
                    *   **流向强制:** 强制内部流向 (如 `DEPOSIT_IN` → `QUARANTINE`)。
                    *   **白名单:** 可选的 `Release` 操作外部目标地址。
                    *   **审批人:** 单一审批组。

    *   **标签:** `Order Id`, `biz_type`
    *   **策略定义:**

## 非功能需求
- 安全性: HSM/MPC, 密钥隔离, 防篡改日志
- 可用性: 高正常运行时间, 故障转移; 目标 RTO/RPO
- 性能: 提现和结算的确定性 SLA
- 可靠性: 幂等操作, 重试机制, 账本一致性
- 可观测性: 日志, 指标, 链路追踪, 异常检测
- 合规性: 证据生成, 定期报告, 鉴证

## 7. 监管与治理

### 合规模式 (仅限 KYB)
*   **KYB (了解你的企业):** 平台专门服务于企业客户 (机构、商户、企业)。因此，**仅要求 KYB**。平台必须在入驻前验证法律实体、UBO (最终受益所有人) 和公司结构。
*   **无 KYC 要求:** 平台不与零售终端用户交互或验证其身份。终端用户的验证是客户机构的唯一责任。

### 治理引擎
*   **策略执行:** 在根、组或子账户级别应用精细化策略 (白名单、速率限制、多签阈值)。
*   **可审计性:** 谁在何时通过什么设备批准了什么的不可篡改日志。

## 集成概览
- **API:** 用于操作 (账户管理、任务创建/列表/状态、资金) 的 REST/GraphQL。
- **事件:** 用于状态变更 (任务状态更新、充值/提现事件) 的 Webhook; 内部事件总线。
- **连接器:** ERP/TMS/核心银行适配器; 基于文件的回退方案。
- **对账:** 提供包含语义标签 (`biz_type`, `biz_id`) 的标准化数据源 (API/文件)，用于第三方账本验证。
